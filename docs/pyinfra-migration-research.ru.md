# Исследование миграции на pyinfra: Salt vs pyinfra для одного хоста

**Дата**: 2026-03-15
**Рекомендация**: **НЕ МИГРИРОВАТЬ** — pyinfra не дает значимого ускорения для masterless-режима на одном хосте и вносит критические регрессии.

## Резюме

| Фактор | Значение |
|--------|----------|
| Ускорение (полный apply) | **~0%** — узкое место в внешних командах (pacman, curl, makepkg) |
| Ускорение (идемпотентный apply) | **~5-6с** быстрее (12с → 6-7с) — только overhead запуска |
| Регрессия (потеря параллелизма) | **+30 мин** на свежем apply (скачивание моделей ollama + video_ai) |
| Трудозатраты на миграцию | **3-5 недель** (35 макросов, 41 файл состояний, 173 вызова макросов) |
| Риск регрессий | **Средне-высокий** — паттерны watch/onchanges, устаревание фактов |
| Bus factor pyinfra | **1** разработчик (Nick Barrett) vs 500+ контрибьюторов Salt |
| Итоговая выгода | **Отрицательная** — затраты значительно превышают пользу |

## 1. Сравнение скорости

### 1.1 Методология

Измерения на основе системы профилирования Salt (`scripts/state-profiler.py --trend`) по 126 логам apply на рабочей станции CachyOS. Оценки pyinfra теоретические (оба инструмента выполняют идентичные внешние команды через subprocess).

### 1.2 Идемпотентный (без изменений) apply

| Фаза | Salt | pyinfra (оценка) | Разница |
|------|------|-------------------|---------|
| Запуск (загрузка модулей, grains) | 5-6с | 1-2с | -4с |
| Компиляция состояний (Jinja+YAML) | 2-3с | 0с (импорт Python) | -2с |
| Выполнение (566 проверок) | 4.3с | ~4-5с | ~0с |
| **Итого** | **~12с** | **~6-7с** | **~5-6с быстрее** |

pyinfra экономит ~5-6с на идемпотентных apply за счёт исключения рендеринга Jinja2 и системы загрузки модулей Salt. Однако эта экономия:
- **Неактуальна для свежих apply** (5-30 минут) — внешние команды доминируют
- **Заметна только когда ничего не меняется** — то есть система уже правильно настроена

### 1.3 Свежий apply с загрузками

| Нагрузка | Время | Узкое место |
|----------|-------|-------------|
| Установка пакетов (pacman -S) | 2-30с на пакет | блокировка pacman, сеть |
| Загрузка бинарников (curl) | 1-10с на файл | Пропускная способность сети |
| Сборка PKGBUILD (makepkg) | 60-3600с на сборку | CPU, компиляция |
| Скачивание моделей (ollama pull) | 60-600с на модель | Сеть, дисковый I/O |
| Итого свежий apply | 5-30 минут | Внешние команды |

Оба инструмента выполняют их через `subprocess.Popen` (Salt `cmd.run` ≈ pyinfra `server.shell()`). Накладные расходы инструмента составляют <1% общего времени. **Нет преимущества в скорости.**

### 1.4 Регрессия параллельного выполнения

Salt поддерживает `parallel: True` для независимых состояний на одном хосте. pyinfra — **нет** (параллелизм только между хостами).

| Параллельное состояние | Файл | Последовательно | Параллельно | Регрессия |
|------------------------|------|-----------------|-------------|-----------|
| `pull_<model>` (5+ моделей) | ollama.sls | ~15 мин | ~5 мин | **+10 мин** |
| `video_ai_download_<model>` | video_ai.sls | ~30 мин | ~10 мин | **+20 мин** |
| `aider_install` | installers.sls | ~30с | ~30с | незначительно |
| `mpv_script_mpris_so` | installers_mpv.sls | ~2с | ~2с | незначительно |
| **Итого** | | | | **+30 мин** |

Это единственное самое разрушительное последствие миграции. В pyinfra нет обходного пути — потребуется реализация кастомных `asyncio`/`threading`-обёрток вне операционной модели pyinfra.

### 1.5 Вердикт по скорости

| Сценарий | Salt | pyinfra | Победитель |
|----------|------|---------|------------|
| Идемпотентный apply | 12с | ~6-7с | pyinfra (+5с) |
| Свежий apply (без параллельных) | 5-10 мин | 5-10 мин | Ничья |
| Свежий apply (с моделями) | 15-20 мин | 45-50 мин | **Salt (+30 мин)** |

## 2. Матрица функциональных пробелов

### 2.1 Сводка

| Серьёзность пробела | Количество | Влияние |
|--------------------|-----------|---------|
| Критическая | 1 | `parallel: True` — нет обходного пути |
| Высокая | 2 | `watch:`/`onchanges:` — многословная замена |
| Средняя | 6 | Граф зависимостей, guards, scoping, built-ins |
| Низкая/Тривиальная | 5 | Управление файлами, retry, загрузка данных |

### 2.2 Детальное соответствие

| Функция Salt | Используется | Эквивалент pyinfra | Пробел |
|---|---|---|---|
| `parallel: True` | 4 состояния | **Нет** (один хост) | **Критический** |
| `watch:` | 4 директивы | `OperationMeta.did_change` | **Высокий** — многословно |
| `onchanges:` | 11 директив | `OperationMeta.did_change` | **Высокий** — многословно |
| `require` / `require_in` | Все состояния | Ручной порядок Python | Средний |
| `creates:` / `unless:` / `onlyif:` | Каждый cmd.run | `_if` callable | Средний |
| scoping состояний | `salt-call state.apply foo` | Отдельные deploy-файлы | Средний |
| `file.serialize` | Изредка | Ручной Python | Средний |
| 400+ модулей состояний | 20+ используются | ~44 операции | Средний |
| `import_yaml` | ~15 файлов данных | `yaml.safe_load()` | Тривиальный |
| `retry:` | Все сетевые состояния | `_retries` + `_retry_delay` | Тривиальный |
| Jinja макросы (35) | 173 вызова | Python функции | Средний (трудозатраты) |
| `file.managed` с `source:` | Широко | `files.template()` | Низкий |
| `service.enabled`/`running` | Широко | `systemd.service()` | Низкий |
| `runas:` | User-сервисы | `_su_user` | Низкий |

### 2.3 Паттерн миграции watch/onchanges

Декларативный `onchanges:` в Salt (3 строки YAML):
```yaml
udev_reload:
  cmd.run:
    - name: udevadm control --reload-rules
    - onchanges:
      - file: udev_rule_file
```

Эквивалент pyinfra (~8 строк Python):
```python
rule_op = files.put(
    name="udev_rule_file",
    src="50-qmk.rules",
    dest="/etc/udev/rules.d/50-qmk.rules",
)
with rule_op.did_change:
    server.shell(
        name="udev_reload",
        commands=["udevadm control --reload-rules"],
    )
```

Этот паттерн нужно применить ко всем 15 директивам watch/onchanges. Версия pyinfra ~в 2.5x многословнее и требует реструктуризации возвращаемых значений макросов для раскрытия объектов `OperationMeta`.

### 2.4 Проблема двухфазной модели

pyinfra собирает факты на этапе "подготовки" и **не обновляет их между операциями**. Это ломает типичные паттерны установка→настройка→запуск:

```python
# ОШИБКА: факт о nginx устарел с фазы prepare
pacman.packages(packages=["nginx"])
files.template(src="nginx.conf.j2", dest="/etc/nginx/nginx.conf")
systemd.service("nginx", running=True)  # Может не сработать — nginx ещё не «известен»
```

У Salt нет этой проблемы — каждое состояние видит состояние системы, оставленное предыдущими состояниями. Обходной путь в pyinfra: разбить deploy на несколько файлов или использовать guards `_if` вместо фактов.

## 3. Оценка трудозатрат на миграцию

### 3.1 Proof-of-Concept порт

`installers.sls` был портирован на pyinfra как репрезентативный data-driven файл состояний.

| Метрика | Salt | pyinfra |
|---------|------|---------|
| Строк кода | 112 (+ ~200 общих макросов) | ~220 (самодостаточный) |
| Определений макросов | 6 макросов в общих файлах | 6 функций inline |
| Загрузка данных | `import_yaml` (1 строка) | `yaml.safe_load()` (3 строки) |
| Guard идемпотентности | `creates:` (1 строка на состояние) | `_if=lambda: ...` (1 строка, многословнее) |
| Параллельные загрузки | `parallel: True` на всех загрузках | Невозможно |
| Отслеживание изменений | `onchanges:` для перезагрузки udev | Безусловная перезагрузка (нет чистого эквивалента) |
| Время порта | — | ~2 часа |

### 3.2 Полная оценка

| Сложность | Файлы | Часов на файл | Итого часов | Риск |
|-----------|-------|---------------|-------------|------|
| **Высокая** | 9 (ollama, video_ai, amnezia, monitoring, dns, services, desktop, fonts, custom_pkgs) | 8-24ч | 72-216ч | Высокий — потеря параллелизма, логика сборки, интенсивное использование макросов |
| **Средняя** | 12 (user_services, kanata, mpd, floorp, installers_desktop, opencode, greetd, steam, network, tidal, installers_themes) | 4-8ч | 48-96ч | Средний — переписывание watch/onchanges |
| **Низкая** | 15 (packages, users, zsh, audio, mounts, sysctl, kernel_*, bind_mounts, snapshots, cachyos*, hardware, llama_embed, system_description) | 1-2ч | 15-30ч | Низкий — простые package/file/service |
| **Библиотека макросов** | 5 файлов (35 макросов) | — | 16-24ч | Низкий — чище как Python |
| **Тестирование** | — | — | 24-40ч | Средний |
| **Итого** | **41 файл** | | **175-406ч (3-5 недель)** | **Средне-высокий** |

### 3.3 Факторы риска

1. **Потеря параллельного выполнения**: Нет пути смягчения в pyinfra. Потребуется кастомная threading-обёртка.
2. **Устаревание двухфазных фактов**: Паттерны установка→настройка→запуск требуют реструктуризации.
3. **Переписывание watch/onchanges**: 15 директив, каждая становится многословнее и подвержена ошибкам.
4. **Уменьшенное количество встроенных операций**: pyinfra имеет ~44 против ~400+ модулей Salt.
5. **Риск единственного мейнтейнера**: Bus factor pyinfra = 1 (см. Раздел 4).

| Метрика | pyinfra | Salt |
|---------|---------|------|
| GitHub stars | ~4,900 | ~15,300 |
| Форки | ~467 | ~5,572 |
| Контрибьюторы | ~30 | 500+ |
| Bus factor | **1** (Nick Barrett) | Команда (VMware/Broadcom) |
| Встроенные модули | ~44 | ~400+ |
| Частота релизов | ~6-8 недель | ~ежеквартально |
| Лицензия | MIT | Apache 2.0 |
| Поддержка pacman | `pacman.packages()` | `pkg.installed` |
| Поддержка AUR | Нет | Нет (оба через макросы) |
| Документация | Хорошая основа, слабое углубление | Обширная, но разрозненная |

**Оценка риска**: pyinfra — здоровый проект с регулярными релизами, но его единственный мейнтейнер создаёт долгосрочный риск. Если Nick Barrett прекратит поддержку, организационной поддержки нет. Salt, несмотря на пренебрежение Broadcom после приобретения, имеет глубокое корпоративное внедрение и большую базу контрибьюторов, что обеспечивает большую устойчивость.

## 5. Рекомендация

### Решение: НЕ МИГРИРОВАТЬ

Все пять критериев оценки провалены:

| Критерий | Порог | Результат | Пройден? |
|----------|-------|-----------|----------|
| Ускорение (свежий apply) | >20% | ~0% | **НЕТ** |
| Регрессия скорости | <5% | +100-200% | **НЕТ** |
| Трудозатраты | <2 недель | 3-5 недель | **НЕТ** |
| Риск экосистемы | Сравнимый или лучше | Хуже (bus factor 1) | **НЕТ** |
| Паритет функций | Без критических пробелов | 1 критический, 2 высоких | **НЕТ** |

### Рекомендуемая альтернатива: оптимизация существующего Salt

Вместо миграции улучшить производительность Salt напрямую:

1. Аудит для дополнительных `parallel: True` — много независимых состояний установки могут работать параллельно
2. Профилирование через `just profile-trend` — выявление реально медленных состояний (главные кандидаты: video_ai_comfyui_setup в среднем 54с, install_opensoundmeter в среднем 3.3с)
3. Предварительная сборка пакетов — кеширование PKGBUILD, которые не менялись
4. Улучшение идемпотентности — замена медленных `unless:` shell-проверок на более быстрые (например, файловые маркеры версий)
5. Оптимизация загрузки модулей Salt — отключение неиспользуемых модулей Salt в конфигурации minion для сокращения времени запуска

### 5.3 Когда pyinfra был бы правильным выбором

pyinfra превосходит в сценариях, которые здесь не применимы:
- **Многохостовое развёртывание** (50+ серверов через SSH) — pyinfra в 5-8x быстрее Ansible
- **Python-native tooling** — когда нужна стандартная Python-линтеризация, тестирование, type checking для CM-кода
- **Встраивание CM в более крупные Python-приложения** — API pyinfra чище для программного использования
- **Greenfield-проекты** без существующих инвестиций в Salt

## Приложение A: Официальные бенчмарки pyinfra

Источник: [Fizzadar/pyinfra-performance](https://github.com/Fizzadar/pyinfra-performance)

| Хосты | pyinfra | Ansible (paramiko) | Ansible (SSH) |
|------:|--------:|-------------------:|--------------:|
| 1 | 1.15с | 3.15с | 4.67с |
| 50 | 4.66с | 25.14с | 40.30с |
| 100 | 8.37с | 48.99с | 78.37с |
| 500 | 49.46с | 256.98с | 416.88с |

**Важное примечание**: Эти бенчмарки измеряют SSH fan-out с тривиальной операцией «проверить, что файл существует». Они не релевантны для однихостового локального выполнения. Salt не включён ни в один бенчмарк pyinfra.
