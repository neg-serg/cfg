# Лучшие практики Salt и интеграция с LLM (2023-2026)

Справочный документ по лучшим практикам управления конфигурацией Salt с фокусом на masterless-установку на одной рабочей станции и разработку с помощью LLM. Каждая практика содержит статус аудита для данного проекта.

**Область**: Salt Project 3006-3008, masterless-режим, CachyOS (на базе Arch)
**Аудитория**: Разработчики, сопровождающие кодовую базу Salt + AI-инструменты (Claude Code, OpenCode)
**Связь с CLAUDE.md**: CLAUDE.md содержит *правила для AI* (предписывающие, краткие). Этот документ объясняет *почему эти правила существуют* и покрывает более широкие темы (здоровье экосистемы, альтернативы, стратегия тестирования). Ссылается на CLAUDE.md по разделам — не дублирует содержание.

**Маркеры статуса**: ✅ Уже следует | ⚠️ Частично следует | ❌ Ещё не внедрено | ➖ Неприменимо

## 1. Организация состояний

### SO-01: Файл-на-домен ✅

**Практика**: Каждый `.sls`-файл отвечает за один домен (audio, fonts, dns и т.д.). Новая функциональность добавляется в существующий доменный файл; новые `.sls`-файлы создаются только когда ни один существующий домен не подходит.

**Обоснование**: Механизм `include` в Salt создаёт неявную связность. Меньше хорошо ограниченных файлов — проще граф зависимостей и понятнее, что затронет `just apply desktop`.

**В проекте**: 44 файла состояний в `states/`, каждый покрывает отдельный домен. Оркестратор `states/system_description.sls` (строки 44-98) включает все домены в логическом порядке.

### SO-02: Макросы для повторяющихся паттернов ✅

**Практика**: Типовые инфраструктурные паттерны (установка пакетов, загрузка с GitHub, управление сервисами) ДОЛЖНЫ использовать макросы из `_macros_*.jinja`. Запрещено дублировать паттерн, уже реализованный макросом.

**Обоснование**: Макросы кодируют устойчивость к сети (retry), защиту идемпотентности (creates/unless), единообразное именование (install_*, build_*) и параллельное выполнение. Обход теряет все четыре гарантии.

**В проекте**: 5 файлов макросов покрывающих все повторяющиеся паттерны:
- `_macros_common.jinja` — общие константы (`retry_attempts=3`, `retry_interval=10`)
- `_macros_github.jinja` — загрузка релизов с GitHub
- `_macros_install.jinja` — установка через curl, pip, cargo, zip, tar
- `_macros_pkg.jinja` — установка через pacman, paru, PKGBUILD, npm
- `_macros_service.jinja` — systemd-сервисы, udev-правила, создание директорий

Когда долгоживущему системному сервису требуется выделенный аккаунт или управляемые пути для runtime/data, предпочитать декларативный инвентарь в `states/data/managed_resources.yaml` плюс общий путь рендеринга `states/systemd_resources.sls` вместо bespoke `user.present`, ad-hoc `file.directory` или одноразовых `tmpfiles.d`-сниппетов.

См. раздел «Macros» в CLAUDE.md для полной справочной таблицы.

### SO-03: Соглашение об именовании State ID ✅

**Практика**: Использовать паттерн `target_descriptor` (например, `loki_config`, `greetd_enabled`). Никогда не использовать пути к файлам как State ID. Исключение: префиксы `install_*` и `build_*` зарезервированы для ID, сгенерированных макросами.

**Обоснование**: Описательные ID делают цепочки `require` читаемыми. Пути в качестве ID ломаются при перемещении файлов и непонятны в логах.

**В проекте**: Последовательно применяется во всех 44 файлах состояний. Параметр `name:` используется явно, когда state ID отличается от целевого пути.

### SO-04: Пороги для встроенного содержимого ✅

**Практика**: Конфиги >= 10 строк — в `states/configs/`, юниты systemd — в `states/units/`, скрипты — в `states/scripts/`. Встроенное содержимое для коротких однострочников.

**Обоснование**: Многострочный YAML в состоянии сложно читать, сложно диффать и невозможно линтировать независимо. Отдельные файлы получают подсветку синтаксиса и могут валидироваться своими нативными линтерами.

**В проекте**: `states/configs/` (шаблоны конфигов), `states/units/` (systemd-юниты), `states/scripts/` (shell-скрипты для `cmd.script`).

### SO-05: Централизованный импорт ✅

**Практика**: Единый файл `_imports.jinja` реэкспортирует все макросы и константы. Файлы состояний импортируют один файл вместо выборки из нескольких файлов макросов.

**Обоснование**: Уменьшает шаблонный код. Когда макрос перемещается между файлами, только `_imports.jinja` требует обновления.

**В проекте**: `states/_imports.jinja` (21 строка) реэкспортирует из `_macros_common.jinja` и предоставляет единую точку импорта для всех `.sls`-файлов.

## 2. Паттерны данных

### DR-01: `import_yaml` вместо pillar в masterless ✅

**Практика**: В masterless-режиме использовать `import_yaml` для всех структурированных данных. Никогда не использовать pillar.

**Обоснование**: Ценность Pillar — шифрование при передаче и таргетирование per-minion через мастер. В masterless minion — единственный потребитель; накладные расходы pillar не дают никакой пользы. `import_yaml` рендерится локально, быстрее и проще для отладки.

**В проекте**: 20+ файлов данных в `states/data/`, загружаемых через `import_yaml`:
- `packages.yaml` — категоризированные объявления пакетов (360+ пакетов)
- `versions.yaml` — зафиксированные версии инструментов
- `installers.yaml` — определения установки CLI-инструментов
- `custom_pkgs.yaml` — локальные PKGBUILD-источники
- `hosts.yaml` — per-host переопределения конфигурации

### DR-02: Конфигурация per-host через grains + host_config ✅

**Практика**: Использовать `grains['host']` для идентификации хоста. Поддерживать центральную карту конфигурации, объединяющую значения по умолчанию с per-host переопределениями. Предоставлять вычисляемые поля (пути, feature-флаги) как единый объект `host` всем состояниям.

**Обоснование**: Grains локальны, быстры и не требуют мастера. Центральная карта конфигурации предотвращает разбросанные по файлам состояний условные выражения.

**В проекте**: `states/host_config.jinja` загружает `data/hosts.yaml`, объединяет дефолты с per-host переопределениями и вычисляет производные поля (`pkg_list`, `primary_output`, feature-флаги). Все состояния ссылаются на `{{ host.* }}` для per-host значений.

### DR-03: Соглашения для файлов данных ✅

**Практика**: Списки пакетов в `states/data/*.yaml` ДОЛЖНЫ использовать формат YAML-списка (один пакет на строку) с инлайн-комментариями, описывающими каждый пакет. Записи отсортированы лексикографически; логическая группировка допустима для больших наборов.

**Обоснование**: Инлайн-комментарии невозможны внутри YAML folded scalars (`>-`). Формат «один на строку» обеспечивает осмысленные диффы, grep-способность и самодокументирование.

**В проекте**: Все файлы данных следуют этому соглашению. Когда файл данных хранит список, потребляемый макросами как строка, шаблон использует `| join(' ')` для восстановления строки через пробел.

См. CLAUDE.md «Data file package lists» и «Package descriptions» для полного описания соглашений.

### DR-04: Граница Salt+Chezmoi с линт-проверкой ✅

**Практика**: Каждый конфиг-файл должен иметь ровно одного владельца — Salt или Chezmoi. Граница проверяется линтером, автоматически обнаруживающим нарушения.

**Правила владения**:
- Salt владеет файлами, требующими: gopass-секреты, service watch-триггеры, не-XDG пути развёртывания, grain-условное развёртывание
- Chezmoi владеет чисто декларативными пользовательскими dotfiles
- Файлы в `dotfiles/`, которые Salt использует через `salt://dotfiles/`, ДОЛЖНЫ быть в `.chezmoiignore`
- Файлы, для которых Salt имеет отдельный шаблон, НЕ ДОЛЖНЫ существовать в `dotfiles/`

**Обоснование**: Двойное владение вызывает конфликты перезаписи без уведомления. Линтер ловит нарушения до того, как они достигнут production.

**В проекте**: `scripts/lint-ownership.py` (87 строк) проверяет эту границу. Запускается как часть `just lint`.

## 3. Устойчивость к сети

### NR-01: Retry на всех сетевых операциях ✅

**Практика**: Каждое состояние, обращающееся к сети, ДОЛЖНО иметь `retry: {attempts: 3, interval: 10}`. Импортировать значения из `_macros_common.jinja` через `_imports.jinja`.

**Обоснование**: Сеть ненадёжна. Временные DNS-сбои, проблемы CDN и лимиты скорости вызывают нестабильные apply. Три повторных попытки с интервалом 10 секунд обрабатывают большинство временных сбоев.

**В проекте**: Все макросы (`curl_bin`, `github_release_system`, `pacman_install`, `paru_install`, `cargo_pkg` и т.д.) включают retry автоматически. Инлайн-состояния `cmd.run`, которые обращаются к сети, должны применять retry вручную.

### NR-02: Параллельные независимые загрузки ✅

**Практика**: Использовать `parallel: True` на независимых состояниях загрузки/установки. Не использовать на состояниях с цепочками `require`.

**Обоснование**: Последовательные загрузки излишне сериализуют сетевой I/O. Типичный apply с 50+ загрузками значительно выигрывает от параллелизма.

**В проекте**: Применяется в `installers.sls`, `installers_mpv.sls`, `ollama.sls` и внутри макросов (`curl_bin`, `github_tar`).

### NR-03: Защита идемпотентности на каждой загрузке ✅

**Практика**: Каждое состояние загрузки/установки ДОЛЖНО иметь `creates:` (файловый маркер) или `unless:` (проверка состояния), чтобы избежать повторного выполнения при каждом apply.

**Обоснование**: Повторная загрузка существующих бинарников тратит пропускную способность и время. Без guard'ов `just apply` занимает 30+ минут вместо секунд при отсутствии изменений.

**В проекте**: Автоматически обеспечивается всеми макросами. См. раздел 4 (Идемпотентность) для рекомендаций по выбору guard.

### NR-04: Флаги curl: всегда `-fsSL` ✅

**Практика**: Все команды curl ДОЛЖНЫ использовать `-fsSL`: `-f` ошибка на HTTP-ошибках, `-sS` тихий режим с показом ошибок, `-L` следование редиректам.

**Обоснование**: Без `-f` curl возвращает exit 0 на страницах 404, молча сохраняя HTML-код ошибки как «бинарник». Без `-L` URL релизов GitHub (которые делают редирект) молча не работают.

**В проекте**: Все макросы используют `curl -fsSL`. Шаблон макроса `curl_bin` в `_macros_install.jinja` (строка 25) устанавливает это как базовое поведение.

## 4. Идемпотентность

### ID-01: Руководство по выбору guard ✅

**Практика**: Каждое состояние `cmd.run` и `cmd.script` ДОЛЖНО иметь guard:

| Guard | Когда использовать | Пример |
|-------|-------------------|--------|
| `creates:` | Состояние создаёт известный файл | `creates: /etc/udev/rules.d/50-qmk.rules` |
| `unless:` | Результат — изменение состояния системы | `unless: rg -qx 'steam' {{ pkg_list }}` |
| `onlyif:` | Состояние зависит от опционального ПО | `onlyif: command -v firewall-cmd` |

**Обоснование**: Guards — основа идемпотентных apply. Без них состояния выполняются при каждом apply, тратя время и рискуя сломать работающую систему.

**В проекте**: См. CLAUDE.md раздел «Idempotency Guards» для полной справочной таблицы.

### ID-02: Предпочитать `creates:` для простоты ✅

**Практика**: Когда состояние создаёт один файл, предпочитать `creates:` вместо `unless:`. Это самый простой и читаемый guard.

**Обоснование**: `creates:` декларативен — Salt проверяет существование файла перед выполнением команды. `unless:` требует написания shell-команды с правильным кодом возврата, что чревато ошибками.

**В проекте**: Широко используется в `dns.sls` (`creates: /var/lib/unbound/root.key`), `installers.sls` (`creates: {{ home }}/.local/bin/aider`) и во всех макросах установки.

### ID-03: Комбинирование `onlyif:` и `unless:` ✅

**Практика**: `onlyif:` и `unless:` можно комбинировать — состояние выполняется только когда `onlyif` успешен И `unless` не пройден. Используйте это для условных состояний, которым также нужна идемпотентность.

**Обоснование**: Некоторые состояния должны выполняться только при наличии опционального ПО (onlyif) И ещё не завершены (unless). Комбинирование guard'ов обрабатывает оба требования.

**В проекте**: Используется в `network.sls` (правила файрвола, условные по наличию `firewall-cmd`).

## 5. Тестирование и валидация

### TV-01: Валидация рендеринга перед apply ✅

**Практика**: Запускать `just validate` (рендерит все состояния, проверяет ошибки Jinja/YAML) перед каждым `just apply`. Ошибка рендеринга в одном состоянии блокирует весь apply.

**Обоснование**: Salt рендерит все включённые состояния перед применением любого из них. Ошибка Jinja-синтаксиса в `fonts.sls` может заблокировать `packages.sls`, даже если они не связаны.

**В проекте**: `just validate` рендерит все 44 состояния и сообщает об ошибках. Занимает ~65 секунд.

### TV-02: Пробный запуск с `test=True` ✅

**Практика**: Использовать `just test STATE` для пробного запуска, показывающего что БУДЕТ изменено без применения. Необходим для просмотра изменений до того, как они затронут живую систему.

**Обоснование**: Живая рабочая станция — нет staging-среды. `test=True` — ближайший аналог preview.

**В проекте**: `just test desktop` показывает, что изменит `desktop.sls`. Доступно для любого отдельного состояния.

### TV-03: Мульти-линтерный конвейер ✅

**Практика**: Запускать несколько линтеров, покрывающих разные аспекты: синтаксис YAML, шаблоны Jinja, shell-скрипты, Python-код, соглашения dotfiles, systemd-юниты, границы владения файлами.

**Обоснование**: Ни один линтер не ловит всё. Комбинация из 8+ линтеров покрывает полный стек от YAML до shell и Python.

**В проекте**: `just lint` запускает: ruff (Python), lint-jinja (кастомный валидатор Jinja, 1178 строк), lint-dotfiles, lint-ownership (граница Salt/Chezmoi), lint-units (systemd), shellcheck, yamllint, taplo (TOML).

### TV-04: Контейнерные smoke-тесты ✅

**Практика**: Тестировать рендеринг и выполнение состояний в изолированном контейнере (Podman + archlinux:latest) перед применением к живой системе.

**Обоснование**: Выявляет ошибки импорта, отсутствующие зависимости и проблемы рендеринга шаблонов без воздействия на реальную систему.

**В проекте**: `tests/smoke-test.sh` (164 строки). Запуск через `just smoke-test`.

## 6. Интеграция с LLM

### LM-01: CLAUDE.md как кодировщик конвенций ✅

**Практика**: Поддерживать полноценный CLAUDE.md в корне проекта со всеми конвенциями, ссылками на макросы, паттернами именования и требованиями к рабочему процессу. Этот файл автоматически загружается в контекст AI при каждом разговоре.

**Обоснование**: AI-инструменты генерируют Salt-состояния на основе обучающих данных с общей документацией Salt. Проектные конвенции (имена макросов, паттерны guard, соглашения о путях) НЕ в обучающих данных. CLAUDE.md восполняет этот пробел, предоставляя проектные правила в каждом разговоре.

**В проекте**: `CLAUDE.md` (343 строки) охватывает: ключевые пути, все 44 модуля состояний с описанием назначения, все макросы с сигнатурами, правила устойчивости к сети, справочник guard идемпотентности, дерево решений cmd.run vs cmd.script, соглашения именования, стиль коммитов, правила владения файлами. Это полнее, чем любой MCP-сервер.

### LM-02: Документация макросов для AI ✅

**Практика**: Документировать каждый макрос с именем, параметрами, назначением и примером использования. Включать значения параметров по умолчанию и ограничения, необходимые AI для генерации корректных вызовов.

**Обоснование**: LLM не могут надёжно читать исходный код Jinja-макросов (вложенный Jinja/YAML сбивает с толку). Явная документация в CLAUDE.md позволяет AI-инструментам использовать макросы корректно без разбора исходников.

**В проекте**: Раздел «Macros» в CLAUDE.md содержит 4 таблицы (по одной на файл макросов), перечисляющие каждый макрос с параметрами и назначением. Примеры использования показаны в разделе «Conventions».

### LM-03: Data-driven паттерны уменьшают ошибки AI ✅

**Практика**: При добавлении пакетов/инструментов/сервисов предпочитать добавление записей в YAML-файлы данных вместо написания новой логики состояний. Файлы данных проще для AI-инструментов — их сложнее изменить неправильно.

**Обоснование**: Добавить строку в `packages.yaml` почти невозможно сделать неправильно. Написание нового состояния `cmd.run` с правильными guards, retry и именованием требует понимания нескольких конвенций одновременно.

**В проекте**: `states/data/packages.yaml` (360+ записей), `states/data/installers.yaml`, `states/data/custom_pkgs.yaml` — AI-инструменты добавляют записи в эти файлы, а не пишут новую логику состояний.

### LM-04: Цикл валидации ✅

**Практика**: После любого изменения, сгенерированного AI, запускать `just validate` для подтверждения, что изменение не сломало никаких состояний. Это закодировано в CLAUDE.md как требование валидации.

**Обоснование**: AI-генерированный Jinja может содержать тонкие ошибки (неправильные имена переменных, отсутствующие импорты, сломанные цепочки require), которые проявляются только во время рендеринга. Цикл валидации ловит их до `just apply`.

**В проекте**: Раздел «Validation Requirements» в CLAUDE.md предписывает запуск `just` (по умолчанию: `system_description`) перед передачей результатов пользователю.

### LM-05: Конституция как ограничители ✅

**Практика**: Поддерживать файл конституции (`.specify/memory/constitution.md`), кодифицирующий нерушимые принципы. AI-инструменты проверяют свою работу на соответствие этим принципам перед представлением результатов.

**Обоснование**: Принципы конституции — это инварианты, которые никогда не должны нарушаться, независимо от запроса пользователя. Они ловят случаи, когда AI-инструмент мог бы «помочь», обойдя проверки безопасности.

**В проекте**: Конституция (версия 1.1.0) определяет 8 принципов: Идемпотентность, Сетевая устойчивость, Изоляция секретов, Макросы в первую очередь, Минимальные изменения, Соблюдение конвенций, Шлюз валидации, Шлюз релиза.

## 7. Анти-паттерны LLM

Типичные ошибки AI-инструментов при генерации Salt-состояний для этого проекта.

### AP-01: Использование pillar в masterless-режиме

**Коренная причина**: Официальная документация Salt акцентирует pillar. LLM переоценивают значимость официальных документов и по умолчанию используют pillar для любого хранения данных.

**До** (неправильно):
```yaml
# pillar/packages.sls — НЕПРАВИЛЬНО для masterless
packages:
  - git
  - tmux

# файл состояния
{% set pkgs = salt['pillar.get']('packages', []) %}
install_packages:
  pkg.installed:
    - pkgs: {{ pkgs }}
```

**После** (правильно):
```yaml
# states/data/packages.yaml
base:
  - git   # система контроля версий
  - tmux  # терминальный мультиплексор

# файл состояния
{% import_yaml 'data/packages.yaml' as packages %}
{{ pacman_install('base_packages', packages.base | join(' ')) }}
```

**Почему**: Pillar требует мастера для рендеринга и шифрования. В masterless-режиме (`salt-call --local`) pillar не даёт преимуществ и добавляет сложность. `import_yaml` локален, быстр и отлаживаем.

### AP-02: Отсутствие guard идемпотентности на `cmd.run`

**Коренная причина**: LLM генерируют состояния `cmd.run`, которые выглядят корректно, но не имеют guard'ов, из-за чего выполняются при каждом apply.

**До** (неправильно):
```yaml
install_tool:
  cmd.run:
    - name: curl -fsSL https://example.com/tool -o /usr/local/bin/tool
```

**После** (правильно):
```yaml
install_tool:
  cmd.run:
    - name: curl -fsSL https://example.com/tool -o /usr/local/bin/tool
    - creates: /usr/local/bin/tool
    - retry:
        attempts: {{ retry_attempts }}
        interval: {{ retry_interval }}
```

**Почему**: Без `creates:` Salt скачивает бинарник заново при каждом apply. Без retry временные сетевые сбои ломают apply.

### AP-03: `file.managed` без `replace: False` для самомодифицирующихся конфигов

**Коренная причина**: LLM не знают, какие инструменты перезаписывают свои конфиги при запуске. Они по умолчанию используют `file.managed`, который перезаписывает при каждом apply.

**До** (неправильно):
```yaml
self_modifying_config:
  file.managed:
    - name: {{ home }}/.config/tool/config.json
    - source: salt://configs/tool.json.j2
    - template: jinja
```

**После** (правильно):
```yaml
self_modifying_config:
  file.managed:
    - name: {{ home }}/.config/tool/config.json
    - source: salt://configs/tool.json.j2
    - template: jinja
    - replace: False
```

**Почему**: Некоторые инструменты перезаписывают свой конфиг при запуске — добавляя умолчания, метаданные, переупорядочивая ключи. Без `replace: False` Salt затирает изменения инструмента при каждом apply, создавая цикл перезапуска. Этот анти-паттерн затронул проект ранее на Telegram bot gateway.

### AP-04: Захардкоженные пути вместо переменных host_config

**Коренная причина**: LLM используют буквальные пути из примеров, не понимая, что пути параметризованы через `host_config.jinja`.

**До** (неправильно):
```yaml
my_config:
  file.managed:
    - name: /home/alice/.config/myapp/config.toml
    - user: alice
    - group: alice
```

**После** (правильно):
```yaml
my_config:
  file.managed:
    - name: {{ home }}/.config/myapp/config.toml
    - user: {{ user }}
    - group: {{ user }}
```

**Почему**: `{{ home }}` и `{{ user }}` импортируются из `_macros_common.jinja` через `_imports.jinja`. Захардкоженные пути нарушают переносимость и принцип DRY.

### AP-05: Создание новых `.sls` вместо расширения существующих доменов

**Коренная причина**: LLM трактуют каждый запрос как изолированный. «Добавить новый сервис» инициирует создание файла вместо расширения соответствующего домена.

**До** (неправильно):
```
# Создание states/my_new_tool.sls для одной pip-установки
```

**После** (правильно):
```
# Добавить в states/installers.sls или states/data/installers.yaml
# так как это установка CLI-инструмента — это соответствующий домен
```

**Почему**: Каждый `.sls`-файл должен быть добавлен в список include `system_description.sls`. Размножение файлов для мелких добавок увеличивает граф зависимостей и нагрузку на сопровождение. Используйте существующие доменные файлы.

### AP-06: Использование pip/npm/cargo вместо макросов pacman

**Коренная причина**: LLM по умолчанию используют языковые менеджеры пакетов. Документация Salt не акцентирует дистрибутивные пакеты.

**До** (неправильно):
```yaml
install_bat:
  cmd.run:
    - name: cargo install bat
    - unless: command -v bat
```

**После** (правильно):
```yaml
{{ pacman_install('bat', 'bat') }}
```

**Почему**: Пакеты Arch/CachyOS предварительно скомпилированы, отслеживаются pacman и автообновляются через `pacman -Syu`. Установки через pip, npm, cargo не отслеживаются, требуют ручных обновлений и могут конфликтовать с системными пакетами. Используйте макросы pacman/paru, если пакета нет в репозиториях или AUR.

### Оценка рисков для данного проекта

**Уровень риска: НИЗКИЙ**

Проект хорошо изолирован от проблем экосистемы Salt:

1. **Только базовые модули**: Использует `file`, `cmd`, `service`, `pkg` — все остаются в ядре Salt. Никаких облачных провайдеров, никаких экзотических execution-модулей.
2. **Masterless-режим**: Нет зависимости от инфраструктуры Salt-мастера, отчётности о заданиях или event bus.
3. **Одна рабочая станция**: Нет управления флотом, таргетирования minion-ов, потребностей в шифровании pillar.
4. **Самодостаточное дерево состояний**: Под контролем версий, без внешних зависимостей от формул.

**Сигналы для наблюдения**:
- Если Salt 3008 LTS задерживается после 2025 → оценить pyinfra как запасной вариант
- Если core-модули (file, cmd, service) перемещаются в extensions → серьёзная эскалация риска
- Если GitHub Discussion #67028 приводит к форку → оценить принятие форка

### Модули, используемые проектом

| Модуль | Категория | Использование |

## 9. Оценка альтернатив

| Инструмент | Сильные стороны | Слабые стороны | Осуществимость миграции |
|------------|----------------|----------------|------------------------|
| **Ansible** | Безагентный (SSH), большое сообщество, поддержка Red Hat, существуют MCP-серверы | Медленнее, нет event bus, только YAML-логика | **Средняя** — Playbook на каждый .sls, Jinja переиспользуется. Высокие трудозатраты на миграцию ролей/коллекций. |
| **pyinfra** | Python-нативный, быстрый, современный дизайн, малый след | Маленькое сообщество, нет экосистемы формул, ограниченная документация | **Средняя** — Pythonic API хорошо отображается на Salt-состояния. Риск: зрелость проекта. |
| **NixOS** | Истинная декларативность, атомарные откаты, воспроизводимость | Крутая кривая обучения, уже мигрировали с Nix | **Низкая** — Повторное принятие маловероятно. Язык Nix фундаментально отличается от модели Salt. |
| **Pulumi** | Настоящие языки программирования, AI-нативный IaC | Облачный фокус, избыточен для рабочей станции, привязка к вендору | **Низкая** — Неподходящий инструмент. Спроектирован для облачной инфраструктуры, не для конфигурации рабочей станции. |
| **Chezmoi (отдельно)** | Лучшее управление dotfiles | Только пользовательские dotfiles, не системные конфиги | **N/A** — Уже используется вместе с Salt. Не может заменить системное управление Salt. |

**Рекомендация**: Оставаться на Salt. Masterless, core-only конфигурация избегает всех зон риска. Если Salt станет неподдерживаемым, pyinfra — наиболее жизнеспособная цель миграции: похожая императивная модель, Python-нативный, активно развивается.
