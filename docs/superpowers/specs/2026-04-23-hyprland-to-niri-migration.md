# Спецификация миграции: Hyprland → Niri

**Дата:** 2026‑04‑23  
**Статус:** Черновик  
**Автор:** opencode  
**Рецензент:** neg

## Контекст

Пользователь использует Hyprland с конфигурацией, ориентированной на производительность и низкую задержку:
- Scrolling‑layout (единственный используемый layout)
- VRR (`variable‑refresh‑rate`)
- Tearing control (`allow_tearing = true`)
- Direct scanout (`direct_scanout = 2`)
- 10‑битный цвет (`bitdepth = 10`)
- Кастомные скрипты (`hypr‑focus‑hist`, `hypr‑win‑list` и др.)
- Множество window‑rules и binds

Цель — оценить целесообразность перехода на Niri, составить план миграции и выявить потенциальные потери/приобретения.

## Цели

1. **Оценить совместимость** текущего workflow с Niri.
2. **Сохранить ключевые функции:** scrollable‑tiling, VRR, 10‑битный цвет.
3. **Минимизировать потерю производительности** (особенно input lag).
4. **Обеспечить плавный переход** с переписыванием конфига и скриптов.
5. **Проверить цветопередачу** (10‑битный цвет должен выглядеть идентично Hyprland).

## Анализ потерь и приобретений

### ❌ Потери (критичные)

1. **`allow_tearing = true`** — в Niri нет аналога. Без этой опции input lag в играх/видео может увеличиться.
2. **`direct_scanout = 2`** — оптимизация рендеринга, отсутствующая в Niri.
3. **Гибкость layout’ов** — Niri предлагает только scrollable‑tiling (другие layout’ы недоступны).
4. **Скрипты и IPC** — все скрипты, использующие `hyprctl` и сокеты Hyprland, требуют переписывания под `niri msg` (JSON‑IPC).
5. **Плагины** — экосистема Hyprland (hyprgrass, hyprfocus, hyprtrails…) в Niri отсутствует.
6. **Синтаксис конфигурации** — полный переписывание конфига с Hyprland‑синтаксиса на KDL.

### ⚠️ Риски

1. **10‑битный цвет** — в Niri нет явной настройки `bitdepth`. Формат пикселей выбирается автоматически через DRM/KMS. Гарантировать идентичный результат нельзя.
2. **Производительность** — отсутствие tearing control и direct scanout может ухудшить игровой/видео‑опыт.
3. **Стабильность** — Niri моложе Hyprland, хотя позиционируется как стабильный.

### ✅ Приобретения

1. **Единообразный scrollable‑tiling** — изначальная философия Niri, без костылей/плагинов.
2. **Динамические workspace’ы с изоляцией мониторов** — окна никогда не «перетекают» на соседний монитор.
3. **Встроенный Overview** — быстрый обзор всех окон и workspace’ов (аналог GNOME Overview).
4. **Встроенный скриншотный UI** — не нужны внешние утилиты (grim).
5. **Улучшенная запись экрана** — блокировка конфиденциальных окон, динамическая смена цели.
6. **Жесты тачпада/мыши** — готовые gesture‑действия без плагинов.
7. **Группировка окон в табы** — организация множества окон.
8. **Градиентные границы с Oklab/Oklch** — современное цветовое пространство.
9. **Анимации с кастомными шейдерами** — возможность писать свои шейдеры.
10. **Xwayland‑satellite** — изолированный процесс для X11‑приложений, повышающий стабильность.
11. **On‑demand VRR** — включать VRR только для определённых окон (игры, видео), снижая артефакты.
12. **Консервативная политика изменений** — обратная совместимость конфига, меньше сюрпризов после обновлений.
13. **Rust‑кодовая база** — меньше багов, связанных с памятью, потенциально выше надёжность.

## План миграции

### Фаза 0: Подготовка (1–2 часа)

1. **Установка Niri** — через AUR (`niri‑bin`) или сборка из исходников.
2. **Создание резервной копии** текущего конфига Hyprland (`~/.config/hypr/`).
3. **Инициализация конфига Niri** — запустить `niri` для генерации default‑config (`~/.config/niri/config.kdl`).

### Фаза 1: Тестирование в windowed‑режиме (2–3 часа)

1. **Запуск Niri внутри Hyprland** — `niri` (без `--session`) для тестирования в окне.
2. **Проверка базовой функциональности** — открытие терминала, навигация по колонкам.
3. **Визуальная проверка цветопередачи** — сравнение градиентов/цветов с Hyprland.
4. **Анализ вывода `niri msg outputs`** — поиск признаков 10‑битного формата.

### Фаза 2: Конвертация конфига (4–6 часов)

1. **Перевод monitor/output настроек** — `monitorv2` → `output "DP‑2" { … }`.
   - Разрешение, частота, scale, положение, VRR.
   - **Проблема:** отсутствие явного `bitdepth`.
2. **Перевод layout‑настроек** — `general`, `decoration`, `dwindle`/`scrolling` → `layout { … }`.
   - Gaps, borders, rounding, blur, shadows.
3. **Перевод input‑настроек** — `input` → `input { … }`.
   - Раскладка, чувствительность, natural‑scroll.
4. **Перевод binds** — `bind` → `binds { … }`.
   - Модификаторы (`$M4` → `Mod`), команды (`exec` → `spawn`).
   - **Особенность:** KDL‑синтаксис, `repeat=false`, `allow‑when‑locked`.
5. **Перевод window‑rules** — `windowrule` → `window‑rule { … }`.
   - Матчинг по классу/тайтлу, `open‑floating`, `geometry‑corner‑radius`.

### Фаза 3: Адаптация скриптов (2–4 часа)

1. **`hypr‑focus‑hist`** — переписать на `niri msg` (подписка на события `window‑opened`/`window‑closed`, команда `focus‑window`).
2. **Другие скрипты** — анализ каждого скрипта, использующего `hyprctl`, и замена на `niri msg action …`.
3. **Интеграция с внешними утилитами** — waybar, rofi, kanshi (если нужно).

### Фаза 4: Полноценный запуск (1 час)

1. **Завершение сессии Hyprland** — выход из DE.
2. **Запуск Niri как сессии** — через display‑manager или `niri‑session`.
3. **Финальная проверка** — все binds, правила, скрипты работают.
4. **Тест производительности** — игры/видео на предмет input lag.

### Фаза 5: Откат (при необходимости)

1. **Возврат к Hyprland** — восстановление резервной копии конфига.
2. **Анализ причин** — если Niri не удовлетворяет по производительности/цветопередаче.

## Критерии успеха

1. **Цветопередача** — 10‑битный цвет работает, градиенты без banding (визуально неотличимо от Hyprland).
2. **Производительность** — приемлемый input lag в играх/видео (субъективная оценка).
3. **Функциональность** — все ключевые binds и window‑rules работают.
4. **Скрипты** — кастомные скрипты адаптированы и функционируют.
5. **Стабильность** — нет критичных багов/падений в daily use.

## Риски и mitigation

| Риск | Вероятность | Влияние | Mitigation |
|------|-------------|---------|------------|
| 10‑битный цвет не работает | Средняя | Высокое | Тестирование в windowed‑режиме перед полным переходом. Если не работает — остаться на Hyprland. |
| Увеличение input lag | Высокая | Высокое | Сравнительный тест в играх. Если неприемлемо — откат. |
| Нехватка функциональности (отсутствие tearing control) | Высокая | Среднее | Принять как компромисс за другие фичи Niri. |
| Сложность конвертации конфига | Средняя | Низкое | Поэтапная конвертация с постоянным тестированием. |
| Баги/нестабильность Niri | Низкая | Среднее | Использовать стабильный релиз (не git‑сборки). |

## Решение

**Рекомендация:** переходить на Niri **только если** готовы пожертвовать tearing control и direct scanout ради стабильности, встроенных фич (Overview, табы, жесты) и изоляции workspace’ов.

**Если производительность и 10‑битный цвет критичны** — остаться на Hyprland.
